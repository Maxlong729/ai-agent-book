# 第13章 从「会做事」到「拥有自己的工作方式」

上一章我们把「几条流程」收拢成了一整套个人工作系统：有盘点、有统一入口、有固定节奏、有稳定嵌入的 Agent。系统搭起来了，接下来要做的事只有一件：让这套方式长在你身上，并开始影响他人。

这一章不再教你「再建一个智能体」或「再画一张表」。我们要聊的是：当工作方式系统化之后，你这个人会怎样变化——思维模式、执行习惯、价值输出。重点在于形成「流程优先」与「方法迁移」的习惯，使个人能力从经验驱动转向系统驱动。最后，我们会回到这本书的起点：工具会更新，但方法论会留下。你带走的不是某一家产品，而是一套可以跟着你走很久的做事方式。

## 13.1 形成新的默认工作方式

系统搭好之后，最容易出现的状况是「偶尔用用」——忙的时候想不起来、闲的时候才打开流程表。要让系统真正生效，需要把「先想流程、再动手」变成你的默认反应，而不是例外。新的默认不是「所有事都流程」，而是：值得重复的事，先有流程再跑。

### 13.1.1 先判断是否要沉淀流程

遇到新任务时，先问自己两个问题：这类事以后还会重复吗？值得为它画一张图吗？如果答案是「会重复」且「值得」，就不要立刻埋头开干，而是先花几分钟想清楚：这件事可以拆成哪几步、每步的输入输出是什么、哪一步可以交给 Agent。想不清楚时，哪怕先列三条「第一步、第二步、第三步」再动手，也会比「这次先做了再说」更有积累。很多人的流程不是一次性设计出来的，而是某次做事时多问了一句「这事能变成流程吗」，然后才慢慢沉淀成一张图、一张表。

### 13.1.2 画流程图后再开始行动

还记得第 3 章说的吗？流程图是结构化工作的起点。哪怕是很简的几步，先有图再执行，和「边做边想」差别很大。先有图，你就知道每一步的接口在哪里、哪里可以接 Agent、哪里必须人把关；边做边想，则容易变成一次性的手工作业。不要求每件事都画成一张标准流程图，但至少养成习惯：遇到重复型任务，先在心里或纸上过一遍「步骤—输入—输出—执行者」，再动工。这个动作本身就会慢慢改变你的默认工作方式：从「接到任务就干」变成「接到任务先想结构」。

### 13.1.3 让流程化成为你的习惯

不追求一次做全、一次做完美。追求的是：每次多问一句——这事能变成流程吗？能记下来下次直接用吗？能交给 Agent 一步吗？问多了，你就会自然地把「可重复」和「可流程化」当成筛选标准，把零散动作一点点收进系统。习惯成自然之后，你不是在「用」一套系统，而是「活」在这套系统里——该跑哪条流程、该在什么时间检查、该把哪一步交给哪个智能体，都会变成条件反射，而不需要每次重新想。

【心法】
新的默认不是「所有事都流程」，而是「值得重复的事，先有流程再跑」。先判断、再画图、再行动，流程化就会慢慢成为你的习惯。

## 13.2 用小动作持续让系统变强

系统搭完不是终点，而是起点。要让系统持续变强，不需要大动干戈，只需要一些小动作、小节奏，长期坚持。

### 13.2.1 保留主流程清单

你在第 12 章已经盘过点、选出了 3～5 条主流程。把这份主流程清单保留下来，当作你的「系统菜单」——随时可查、可增删。新加一条流程就补进清单，某条流程很久不用就标成「暂停」或移出主菜单，但不要丢。主流程清单是你个人工作系统的目录，有了它，你就不会忘记「自己还有哪些流程可用」，也不会在忙乱时又退回「想到啥做啥」的旧模式。可以把它放在统一入口的显眼位置，或单独建一页「我的主流程」，每周检查时扫一眼，确认每条流程的状态。

### 13.2.2 做一次五分钟复盘

每周流程检查时（第 12 章 12.3 的固定节奏），多问一句：这条流程里，有没有一步可以交给 Agent？有没有一步可以合并或删掉？有没有一步总是卡住、需要改一改？不需要立刻改，只要把答案记下来，放进「待优化清单」。例如：发现「写周报时每次都要自己汇总零散记录」，就记一条「周报流程—汇总步骤—可交给 Agent」；发现某步总在等别人反馈才推进，就记「优化接口或提前沟通」。每次只花五分钟，但长期积累下来，你会有一张清晰的「系统可改进点」列表，下次有空时按清单逐条优化即可。系统变强，靠的是这种小步复盘，而不是某一天推倒重来。

### 13.2.3 记录可交给 Agent 的步骤

在做事或复盘时，一旦发现「这一步好像可以交给 Agent」，就记下来：哪条流程、哪一步、打算用什么类型的 Agent（写作类、信息类、图像类等）。不要求立刻去扣子编程上搭，先记下来就行。积累成「待交给 Agent 的步骤」清单，每月或每季度挑一两步落实，系统就会一点点「变聪明」。这样你既不会因为「想一次做完」而拖延，也不会忘记那些本可以自动化的环节。

### 13.2.4 每月优化一条流程

不贪多。每月只动一条流程：要么增加一步 Agent、要么简化一步、要么调整顺序或接口。动完就稳下来用一阵，看效果再决定下一步改什么。系统变强靠的是小步迭代，不是推倒重来。每月一条，一年就是十二条优化，足够让一套个人工作系统脱胎换骨。

【心法】
系统变强靠的是「小步迭代」——保留主流程清单、五分钟复盘、记录可交给 Agent 的步骤、每月优化一条。不需要某一天大改，只需要持续小改。

## 13.3 让你的工作方式开始影响别人

当你的流程和系统稳定运转之后，它们会自然开始「外溢」：同事会问你「这事你怎么做得这么快」，合作方会注意到「和你对接特别清晰」，新人会按你的流程上手。这时，工作方式就不再只是你个人的效率工具，而成了你的价值输出——别人看得见、学得会、协作得上。

### 13.3.1 用流程回应同事提问

当同事问你「这事怎么做」「你们一般怎么走」时，不要只口头讲一遍。直接给出一张流程图或一份步骤清单：「我们一般这么走：第一步……第二步……这里用 Agent 出初稿，这里人把关。」对方一眼能看懂，你也不用每次重复解释。用流程回应提问，既节省你的时间，也把「我们团队/我们部门是怎么做事的」沉淀下来，新来的人也能按图索骥。例如：别人问「你们会议纪要怎么出的」，你发过去一份「会议纪要流程」——录音转文字稿丢进纪要助手、出草稿后人补重点与待办、按模板分发；对方照做就能对齐你的节奏。

### 13.3.2 用流程推进合作事项

跨部门、跨团队协作时，最容易出现「你以为我说清楚了、我以为你听懂了」的偏差。用流程推进合作：把协作拆成有步骤、有输入输出、有节点的节奏，对方更容易跟上，你也更容易检查「卡在哪一步」。例如：合作方需要你提供什么、你在什么时间前给、对方在什么时间前反馈，写进一张简表或一条简短流程，大家按同一套节奏走，协作成本就会明显下降。你甚至可以把自己在 12 章建立的「统一入口」思路推广到协作：约定一个共享的协作入口（如一份共享文档、一个任务看板），把「待对方提供」「待我方反馈」标清楚，流程化协作本身也会成为你的影响力。

### 13.3.3 用可复用的方法带团队成长

还记得第 3 章讲的可交接性吗？当你的流程和 Agent 写进了文档、写进了流程版本，新人就可以按流程上手，而不必完全依赖你口传心授。你可以把主流程清单、关键步骤的输入输出、以及「哪一步用哪个智能体」打包成一份「新人上手指南」，让团队里更多人用同一套方式做事。这样你带的不只是「人」，而是「可复制的工作方式」——人会流动，方式可以留下。

### 13.3.4 让方式方法成为你的价值输出

你的价值不仅是「做了啥」，还有「你是怎么做的、别人能不能复用」。当你的流程和系统能被别人看懂、复用，你就从「做事的人」多了一个「设计工作方式的人」的维度。这在职场里会逐渐变成一种可辨识的能力：这个人不仅能把事做完，还能把做事的方法沉淀下来、传下去。工作方式系统化之后，这种能力会自然显现。

【心法】
当你的流程和系统能被别人看懂、复用，你就从「做事的人」多了一个「设计工作方式的人」的维度。用流程回应提问、用流程推进合作、用可复用方法带团队，方式方法就会成为你的价值输出。

## 13.4 工具会更新，但方法论会留下

这本书里我们用了扣子编程做实操、用五类人群做了案例，但你要带走的不是某一家产品，而是一套可迁移的做事逻辑。工具会更新、软件会迭代，但方法论会留下。

### 13.4.1 工具在变，做事逻辑不变

五步法、流程图、输入输出、人机分工——这些逻辑换一个平台一样用。不管将来你用的是扣子、Dify、n8n 还是还没诞生的新工具，只要你心里有「明确目标、拆解步骤、定义接口、分配执行者、沉淀资产」这套逻辑，你就能在新平台上快速复现你的工作系统。工具在变，做事逻辑不变。

### 13.4.2 软件可换，流程也可迁移

流程是抽象的结构：步骤顺序、每步的输入输出、哪一步由人做、哪一步由 Agent 做。这些信息可以写在一份文档里、一张表里，与具体软件解耦。今天你在扣子编程上搭的智能体，明天你可以用同一份流程描述在别的平台上再搭一遍；今天用的待办 App，明天可以换一个，只要「统一入口、标注流程、按节奏检查」的规则不变，系统就能迁移。软件可换，流程也可迁移。

### 13.4.3 留下的是方法，不是技巧

技巧会过时——某款产品的按钮怎么点、某个模型的参数怎么调，可能明年就变了。但方法会长期有用：如何把任务拆成步骤、如何定义人与 Agent 的边界、如何用节奏和复盘让系统持续进化、如何把流程写成可交接的文档。这些方法不依赖某一家公司、某一款产品，它们是你自己的「工作方式操作系统」。留下的是方法，不是技巧。

### 13.4.4 你的「工作方式 2.0」开始了

回忆一下前言里我们讲过的「手工作坊」和「智能工厂」：在手工作坊里，你每次接到任务都从零开始、凭感觉干；在智能工厂里，你有了图纸、有了流水线、有了一批按角色分工的「数字员工」。读完这本书、搭完你的流程和系统之后，你已经有了一张属于自己的「工厂图纸」和一批可用的 Agent。接下来，就是持续运转、迭代、让这套方式长在身上、并开始影响他人。

这不是终点，而是起点。工具会继续更新，你可以随时把新平台、新模型接进原有流程；流程也可以继续增删，你的主流程清单会随着工作重心变化而演化。唯一不变的，是你已经建立起来的「先想结构、再分配执行者、再用节奏和复盘保持系统运转」的思维习惯。你的「工作方式 2.0」已经开始了。

## 全书结语

写到这里，这本书就要和你告别了。

我们一起走了很长一段路。从厘清 AI、Agent 与工作流的关系开始，到用流程图和五步法把想法变成可执行的流水线，再到在扣子编程上亲手搭出选题、脚本、会议纪要、方案分析……一批真正能用的智能体；最后，把零散的流程收拢成个人工作系统，并聊了系统搭好之后，你这个人会怎样变化——新的习惯、小步迭代、影响他人，以及为什么「工具会更新，方法论会留下」。如果你曾跟着某一章、某一条流程动手做过，哪怕只做过一次，你已经不再是那个只能凭感觉做事的人了。你心里有了一张图，手里有了一批数字员工，你知道「先想结构、再分配、再用节奏保持运转」是怎么一回事。

前言里我说过，希望你能完成三层进化：掌握一套通用的工作流方法论、学会给 Agent 写岗位说明书、建立属于自己的个人智能工作系统。这些不是空话——它们会跟着你走。换一个平台、换一份工作，只要你愿意把「值得重复的事先变成流程」当成习惯，这套方法就会一直在。你带走的，不是某一家产品，而是一套可迁移的做事方式、一批可复用的流程与智能体，以及一种新的默认：先想流程，再动手。

从手工作坊到智能工厂，你已经拿到了自己的图纸。接下来，就在日常工作中反复用、持续练就好。

最后顺便提一句：眼下不少平台（如扣子 2.0）正在推「Skill（技能）」——把某一类能力封装成可复用模块，供多个智能体配置或单独调用。你可以理解为：智能体像你招来的一个员工（有名字、有岗位，你跟他对话、给他派活）；Skill 像公司里谁都能领用的一套标准工具或操作手册（比如「查订单」「生成报表」），同一套可以配给不同的员工，也可以单独拿出来用。二者互补，本书专注教你搭智能体与工作流，Skill 作为平台新方向，留待你在用熟智能体之后再去探索即可。

感谢你一路读到这里，愿这套方法在你手中越用越熟、越用越稳。
