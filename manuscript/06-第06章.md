# 第6章 让流程进入日常：从试跑到稳定执行

完成了流程图的绘制与提示词的撰写，并不意味着工作流的建设已经结束。
**恰恰相反，真正的考验才刚刚开始。**

许多读者的 Agent 之旅往往止步于此：兴致勃勃地设计了一个看似完美的方案，一经运行却频频报错，反复调试无果后，最终得出结论：「这个工具并不实用。」

事实上，问题往往不在于工具本身，而在于**「上线」**的方式。在软件工程领域有一条铁律：**没有任何程序是第一次运行就能完美无缺的。** 你的 Agent 工作流也不例外。

这一章，我们不谈设计，谈**「调试」**。怎么把一个「纸上谈兵」的方案，变成一个**「经得起实战」**的生产力工具。

## 6.1 用正确方式试跑流程的第一版

**❌ 错误反应：**
「我要把所有细节都想清楚，把所有 prompt 都写得无懈可击，然后再点击运行。」

**💥 后果：**
你试图一步到位，结果反而陷入停滞。
因为你预想的 100 种情况，现实中只出现了 1 种，而现实中出现的第 101 种情况（比如断网、API 报错、Token 超限），你根本没预案。

**✅ 正确逻辑：**
**先跑通，再跑对，最后跑快。**
V1 版本的唯一使命，就是**「能跑完」**。哪怕中间结果是错的，只要能从头走到尾，就是胜利。

**📝【MVP 试跑三原则】**

| 原则 | **解释** | **错误示范** | **正确示范** |
| :--- | :--- | :--- | :--- |
| **低风险启动** | 别拿老板明早要看的报告试手。 | 直接用工作流写正式邮件发给客户。 | 先发给自己这台电脑，确认无误再转发。 |
| **全量数据减半** | 数据越多，出错概率越大。 | 一次性让 Agent 处理过去 3 年的 Excel 表。 | 先切 10 行数据喂给它，通了再加。 |
| **不仅看结果** | 过程比结果重要。 | 只看最后生成的报告对不对。 | 盯着每一步的输出，看它是在哪一步跑偏的。 |

**【心法】**
不要在这个阶段追求完美。**V1 就是用来报错的。** 每一个报错，都是 Agent 在告诉你它的边界在哪里。

## 6.2 为关键步骤设置必要的检查点

很多惨案（比如 AI 给客户发脏话），都是因为在这个环节失守了。

**❌ 错误反应：**
「既然自动化了，那我就不管了，全权委托。」

**💥 后果：**
这就好比你雇了个新员工，第一天就让他代表公司去签合同，还不审核条款。这不是信任，这是**渎职**。

**✅ 正确逻辑：**
**信任，但要验证（Trust, but verify）。**
你需要在工作流的关键路口，设置**「人工安检站」**。

**📝【红绿灯检查机制】**

*   **🟢 绿灯区（Fully Auto）**：
    *   **定义**：出错了也没什么大不了，或者后续步骤能自动修正。
    *   *比如：搜集新闻素材、整理会议录音、个人笔记归档。*
    *   **策略**：完全放权，每周抽查一次即可。

*   **🟡 黄灯区（Human Review）**：
    *   **定义**：需要对外输出，但主要是内容层面的。
    *   *比如：写公众号草稿、生成设计图、给同事的周报。*
    *   **策略**：**AI 生成 -> 人工修改 -> 发送**。你必须保留「修改权」。

*   **🔴 红灯区（Human Confirm）**：
    *   **定义**：涉及金钱、法律、核心人脉，一旦出错不可挽回。
    *   *比如：给大客户报价、自动转账、发布正式公关稿。*
    *   **策略**：**AI 准备草稿 -> 必须由人工点击「确认」按钮 -> 才能执行。**

**【心法】**
哪怕 Agent 准确率达到 99%，如果你为了省那 1% 的检查时间而导致了一次「严重失误」，也是得不偿失。**检查点不是累赘，是安全带。**

## 6.3 建立流程的出错日志与复盘机制

**❌ 错误反应：**
「哎呀又错了！」（然后默默手动改过来，掩耳盗铃）

**💥 后果：**
同一个坑，你今天踩了，下周还会踩。你的工作流永远停留在「如果不看着就会出错」的婴儿阶段。

**✅ 正确逻辑：**
**把错误当成养料。**
每一次 Agent 出现错误，都是因为它得到的指令不够清晰，或者背景信息不够多。记录下来，反哺给你的 Prompt。

**📝【Bug 捕获日志】**

当你发现 Agent 结果不对时，不要只是生气，填这张表：

| 错误现象 | **我的指令 (Prompt)** | **AI 的理解偏差** | **修正方案** |
| :--- | :--- | :--- | :--- |
| *生成的摘要太长了，有 500 字。* | *「请帮我总结一下这篇文章。」* | *AI 以为你要详细总结，因为它不知道「短」的标准。* | *增加约束：「请总结成 3 个要点，总字数不超过 100 字。」* |
| *推荐的竞品太老旧。* | *「找一下竞品。」* | *AI 搜索了历史数据，不知道你要最新的。* | *增加约束：「仅搜索最近 3 个月的新闻。」* |

**【心法】**
Agent 就像你的徒弟。它犯错不可怕，可怕的是你作为师父**只知道骂它笨，却从来不教它**。

## 6.4 把流程从 v1 升级到 v3 的演进路径

最后，给你的预期做一个管理。一个好用的工作流，通常要经历三个版本。

*   **v1.0：婴儿期（能跑通）**
    *   **状态**：磕磕绊绊，经常报错，需要你盯着看。
    *   **目标**：把流程跑完，别中断。

*   **v2.0：成长期（更稳定）**
    *   **状态**：大部分情况能对，偶尔遇到「边缘情况」（Corner Case）会失效。
    *   **目标**：把 Prompt 写得更严谨，堵住漏洞。

*   **v3.0：成熟期（更高效）**
    *   **状态**：稳定运行。你已经敢把它放进后台运行了。
    *   **目标**：优化成本（换更便宜的模型），优化速度（并发执行）。

**【心法】**
别指望一步登天。**好用的 Agent 是「调」出来的，不是「写」出来的。**
