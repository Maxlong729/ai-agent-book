**【样章】第5章 给 Agent 写一份清楚的"岗位说明书"**

**5.1 用四个要素定义一个 Agent 的岗位**

这节只解决一件事：让 Agent 不再"每次都像换了个人"，减少你反复补充要求、返工的次数。

想象一下，如果你对新来的实习生说一句"去帮我把周报写了"，他大概率会给你一堆你没法用的废话。你肯定会骂他笨，但责任其实在你------因为你没告诉他写给谁看、用什么格式、重点突出发哪个数据。

在 Agent 的世界里，没有"默契"这回事，只有"定义"。也就是说，你不缺更长的提示词，你缺的是一份能长期使用的"岗位说明书（JD）"。

一份合格的岗位说明书不需要写成作文，只要把这四个要素卡死就够了：工作目的、输入来源、输出成果、限制条件。 少一个，问题就会在对应位置爆出来。

接下来，我们用同一个例子（周报 Agent）讲透这四个要素。

**5.1.1 明确 Agent 的工作目的**

❌ 错误写法：

"帮我写一份周报，要专业一点，有深度一点。"

后果**：** Agent 会开始堆砌术语，因为它理解的"专业"就是不说人话。

✅ 正确逻辑：

工作目的只回答一个问题：这份工作到底要替你解决什么具体问题？

不要写形容词，要写"动作"和"对象"。把三件事说清楚：给谁看？用来干嘛？什么叫合格？

**📝【通用模板】工作目的**

+:-------------------------------------------------------------------------------------------+
| 你是 【岗位名称】。                                                                        |
|                                                                                            |
| 你的工作是把我提供的 【原始材料】，整理成一份可直接发送给 【阅读对象】 的 【交付物名称】。 |
|                                                                                            |
| 这份文件的核心用途是： 【用途 A、用途 B】。                                                |
|                                                                                            |
| 合格标准是：【标准 1】、【标准 2】。                                                       |
+--------------------------------------------------------------------------------------------+

**【周报 Agent 示范】**

+:-------------------------------------------------------------------------------------------------+
| 你是【周报 Agent】。                                                                             |
|                                                                                                  |
| 你的工作是把我提供的【本周要点和数据】，整理成一份可直接发送给【直属领导与协作同事】的【周报】。 |
|                                                                                                  |
| 这个周报的用途是：【同步进展、暴露风险、明确下周诉求】。                                         |
|                                                                                                  |
| 合格标准：【四个板块齐全、条目清晰、数据可核对】。                                               |
+--------------------------------------------------------------------------------------------------+

**5.1.2 告知 Agent 的输入来源**

❌ 错误写法：

"根据你的知识，写一份周报。"

后果**：** Agent 会用"常识"去编造细节。周报最怕的不是写得难看，而是"看起来很顺畅，但事实全是编的"。

✅ 正确逻辑：

输入来源只回答一个问题：你允许它用哪些材料？

必须明确这一条规则：你给它什么，它就只能用什么。你没给的，它必须停下来问，绝对不能自己补。

**📝【通用模板】输入来源**

+:------------------------------------------------------------------------------------------+
| 你只能使用以下输入：                                                                      |
|                                                                                           |
| \[输入材料 A\]；                                                                          |
|                                                                                           |
| \[输入材料 B\]；                                                                          |
|                                                                                           |
| \[输入材料 C\]。\                                                                         |
| 红线： 不得自行补充事实，不得自行搜索或引用外部信息。遇到信息缺失，必须先列出"问题清单"。 |
+-------------------------------------------------------------------------------------------+

**【周报 Agent 示范】**

+:-------------------------------------------------------------------------------------------------+
| 你只能使用以下输入：① 我粘贴的本周工作要点；② 我提供的关键数据与截图；③ 我提供的项目名与负责人。 |
|                                                                                                  |
| 红线： 严禁自行脑补数据。                                                                        |
+--------------------------------------------------------------------------------------------------+

**5.1.3 规定 Agent 的输出成果**

❌ 错误写法：

"整理一下发给我。"

后果**：** 它一会给你写成大段文字，一会给你写成表格，你每次都要花 10 分钟重新排版。

✅ 正确逻辑：

输出成果只回答一个问题：你要它交付的成品长什么样？

最好的输出描述不是"写清楚"，而是"定字段"。把输出看作是一个填空题，每个空填什么，必须规定死。

**📝【通用模板】输出成果**

+:----------------------------------------------------------------------+
| 你的输出必须严格遵守以下结构：                                        |
|                                                                       |
| 板块 1 \[标题\]： 包含 \[要素 A + 要素 B\]，限制 \[X\] 条；           |
|                                                                       |
| 板块 2 \[标题\]： 包含 \[要素 C\]；                                   |
|                                                                       |
| 板块 3 \[标题\]： 包含 \[要素 D\]。                                   |
|                                                                       |
| 文风要求： 不写背景科普，不写空话套话，不用模糊词（如"可能、大概"）。 |
+-----------------------------------------------------------------------+

**【周报 Agent 示范】**

+:-------------------------------------------------------------+
| 你的输出固定为四个板块：                                     |
|                                                              |
| 本周完成： 3 条，每条包含【结果 + 数据 + 影响】；            |
|                                                              |
| 关键进展： 1 条，说明【进度 + 下一步】；                     |
|                                                              |
| 风险与卡点： 1 条，说明【原因 + 需要的支持】；               |
|                                                              |
| 下周计划： 3 条，每条包含【动作 + 截止时间 + 协作方】。      |
+--------------------------------------------------------------+

**5.1.4 标注 Agent 的限制条件**

❌ 错误写法：

很多人根本不写这一条。

后果**：** 越用越不敢用，因为你不知道它什么时候会给你埋个雷，比如替你承诺了根本做不到的完成时间。

✅ 正确逻辑：

限制条件只回答一个问题：哪些事它绝对不能替你做主？

这一部分是 Agent 的"保命符"。写清楚了，你才敢把任务放心交出去。

**📝【通用模板】限制条件**

+-----------------------------------------------------------------------------------------------+
| 不承诺： 不替我做任何承诺性表述（如"保证完成"、"一定"）；                                     |
|                                                                                               |
| 不归责： 不替我判断责任归属，只陈述事实；                                                     |
|                                                                                               |
| 不推测： 涉及数据、时间、金额，必须逐项引用我提供的来源，不得推测；                           |
|                                                                                               |
| 兜底机制： 遇到目标冲突或信息不足，先给"问题清单"，再给"保守版草稿"，并标注需要我确认的句子。 |
+-----------------------------------------------------------------------------------------------+

**【周报 Agent 示范】**

+:-------------------------------------------------------------+
| 限制条件：                                                   |
|                                                              |
| 不使用"确保"、"肯定"等绝对化词汇；                           |
|                                                              |
| 任何数据必须有我提供的原始出处；                             |
|                                                              |
| 如果我没提供下周计划，不要自己编，直接留白并标注"待确认"。   |
+--------------------------------------------------------------+

**5.2 为不同类型的 Agent 设计不同的岗位**

同样是"岗位说明书"，找资料的、做分析的、写文章的，重点完全不同。

很多人的 Agent 不好用，是因为你总想招一个"全能助理"------既能像记者一样查资料，又能像咨询顾问一样给建议，还能像文案一样写爆款。结果通常是：它什么都干了，但什么都没干好。 一会儿像百度百科，一会儿像地摊文学。

这一节你只记住一句话：先看你要的"最终交付物"是什么，再决定岗位类型。交付物是资料包？是判断结论？还是发布文案？下面这三类最典型的岗位，请直接照着抄。

**5.2.1 信息类 Agent：负责搜集与整理**

😭 典型翻车现场：

你让它"帮我找找最近有什么 AI 热点，顺便写个短评"。

结果： 它给你编造了几个不存在的新闻，还一本正经地评论了一番。你拿去发朋友圈，直接被打脸。

🛑 管理者思维：

信息类 Agent 只做一件事：把你写稿需要的材料找齐、理顺、标清来源。

它交付的不是观点，更不是成稿，而是一份你拿来就能用的"干净资料包"。死规矩： 哪怕找不到，也不许编。

**📝【通用模板】信息类 Agent**

+:---------------------------------------------------------------------------------+
| **【岗位名称】** **\[主题\]** 资料搜集 Agent                                     |
|                                                                                  |
| **【工作任务】**为我提供一份"可核对的资料包"，不输出观点，不写脚本，只陈述事实。 |
|                                                                                  |
| **【检索范围】**                                                                 |
|                                                                                  |
| **时间：** 仅限近 **\[X\]** 天；                                                 |
|                                                                                  |
| **来源：** 仅限 **\[指定网站/权威媒体\]**；                                      |
|                                                                                  |
| **排除：** 排除广告软文、无来源的自媒体转载。                                    |
|                                                                                  |
| **【交付格式】**输出必须包含两部分：                                             |
|                                                                                  |
| A 部分 - 原始摘录： 每条摘录必须保留【原文关键句 + 来源链接 + 发布时间】；       |
|                                                                                  |
| B 部分 - 结构化清单： 按主题归类（如"用户痛点"、"行业数据"），每类不超过 5 条。  |
|                                                                                  |
| 【红线】如果找不到可靠来源，必须明确写"未找到可核对材料"，不得用猜测补齐。       |
+----------------------------------------------------------------------------------+

**5.2.2 分析类 Agent：负责判断与结论**

😭 典型翻车现场：

你把一堆杂乱的会议记录丢给它："帮我分析一下。"

结果： 它给你输出了一堆正确的废话，比如"我们要加强沟通"、"我们要提高效率"。看起来很完整，实际毫无执行价值。

🛑 管理者思维：

分析类 Agent 的价值不在于"写得像"，而在于"判断站得住"。

它必须基于你给的事实，通过你给的逻辑，推导出结论。死规矩： 所有结论必须有据可依，不能凭空"觉得"。

**📝【通用模板】分析类 Agent**

+:---------------------------------------------------------------------------------------+
| **【岗位名称】** **\[任务\]** 深度分析 Agent                                           |
|                                                                                        |
| **【输入限制】**你的分析依据只来自：                                                   |
|                                                                                        |
| 我提供的 **\[具体材料/数据\]**；                                                       |
|                                                                                        |
| 我给定的 \[判断标准/原则\]。\                                                          |
| 注意： 超出材料范围的事实，一律不许引用。                                              |
|                                                                                        |
| **【交付格式】**你的输出必须包含三个维度：                                             |
|                                                                                        |
| **核心结论：** 用 3 个短句概括；                                                       |
|                                                                                        |
| **判断依据：** 每个结论下方，必须引用原文中的具体句子或数据作为支撑；                  |
|                                                                                        |
| **风险提示：** 指出哪些结论因信息不足而可能存在偏差。                                  |
|                                                                                        |
| **【红线】**遇到信息缺失导致无法判断时，先列出"我需要补充的问题清单"，不要强行下结论。 |
+----------------------------------------------------------------------------------------+

**5.2.3 写作类 Agent：负责内容产出**

😭 典型翻车现场：

你给它一个标题："帮我写篇关于咖啡的文章。"

结果： 它写了一篇"咖啡的历史、种类、好处"的百科全书。文笔没问题，但根本没人看，因为没有观点，没有情绪，也没有对象。

🛑 管理者思维：

写作类 Agent 的职责很单一：把你已经确定的要点，填进固定的结构里。

它不负责找资料，也不负责想观点。它只负责"戴着镣铐跳舞"。死规矩： 结构第一，文采第二。

**📝【通用模板】写作类 Agent**

+:--------------------------------------------------------------------------------------+
| **【岗位名称】** **\[平台/类型\]** 写作 Agent                                         |
|                                                                                       |
| **【输入投喂】**我会给你提供：                                                        |
|                                                                                       |
| **核心观点/选题：** \[一句定调\]；                                                    |
|                                                                                       |
| **素材要点：** \[清单\]；                                                             |
|                                                                                       |
| **目标读者：** \[谁？想听什么？\]。                                                   |
|                                                                                       |
| **【风格设定】**                                                                      |
|                                                                                       |
| 请模仿 \[风格关键词，如：毒舌、温情、干练\] 的口吻。                                  |
|                                                                                       |
| 句式要求： 多用短句，少用形容词，禁止使用"总而言之"、"综上所述"等 AI 味十足的连接词。 |
|                                                                                       |
| **【交付结构】**                                                                      |
|                                                                                       |
| **开头（3秒钩子）：** 用一个\[反直觉事实/提问\]开场；                                 |
|                                                                                       |
| **正文（黄金三点）：** 分三段论述，每段必须有一个\[具体案例/数据\]；                  |
|                                                                                       |
| **结尾（行动呼吁）：** 引导读者进行\[某个动作\]。                                     |
+---------------------------------------------------------------------------------------+

**5.3 如何判断任务需不需要拆成多个 Agent**

很多读者问我："能不能把搜集、分析、写作、排版全塞进一个 Agent 里？这样省事。"我的回答永远是：你可以这么做，但它一定会变笨。你决定要不要拆 Agent，不是看任务大不大，而是看"脑子顺不顺"。

想象一下，你让一个厨师一边切菜，一边炒菜，还要一边去菜市场买葱。他一定会手忙脚乱，最后把葱买错了，或者把菜炒糊了。Agent 也是一样。当它在"发散找资料"的时候，你就不能强迫它同时"收敛下结论"。

怎么快速判断？请使用下面这个"关键词检测法"：

+:---------------------------------------------------------------------+
| 检查你的"工作目的"描述。如果你不得不使用以下连接词，**请立刻拆分**： |
|                                                                      |
| **🚩 必须拆分的信号词：**                                            |
|                                                                      |
| **"然后"**（代表时间先后，需要接力）                                 |
|                                                                      |
| **"并且"**（代表多线程，容易干扰）                                   |
|                                                                      |
| **"顺便"**（代表注意力分散，质量必降）                               |
|                                                                      |
| **"同时"**（代表角色冲突，逻辑打架）                                 |
|                                                                      |
| ❌ 错误指令： "去搜集资料，然后分析一下，顺便写个周报。"             |
+----------------------------------------------------------------------+

✅ 正确拆解： \[搜集员\] -\> \[分析师\] -\> \[撰稿人\]。

**5.3.1 将"搜集"与"判断"拆分：避免"自圆其说"**

🛑 为什么必须拆？

如果你让同一个 Agent 既负责"找证据"，又负责"下判决"，它会天然倾向于"只找能证明自己观点的证据"。这就叫"认知混流"。

比如你问它："喝咖啡对身体有害吗？"它如果先找到了"有害"的文章，后面的分析就会全是负面的。你以为你得到了客观结论，其实你得到的是它的"偏见回音壁"。

**🛠️ 拆分方案：**

Agent A（资料员）： 只管把正反两面的信息都搬回来，不许说话，不许评价。

Agent B（法官）： 看着 A 搬回来的材料，进行客观判决。

**📝【拆分示范】**

+:----------------------------------------------------------------+
| \[资料员\] 输出： 10 条原始数据（5 条利好，5 条利空），纯陈述。 |
|                                                                 |
| \[分析师\] 输入： 上面那 10 条数据。                            |
|                                                                 |
| \[分析师\] 输出： "综合来看，风险大于机会，依据是数据 A 和 B。" |
+-----------------------------------------------------------------+

**5.3.2 将"写作"与"改写"拆分：治疗"精神分裂"**

🛑 为什么必须拆？

写作需要"发散"，改写需要"克制"。

如果你让 Agent "写一篇那种很爆款、很严谨、字数又很少的文章"，它会崩溃。因为它刚写了一句觉得不够爆款，想改；改完觉得不够严谨，又改。

最后的结果通常是：四不像。 既没有干货，也不好笑，充满了正确的废话。

**🛠️ 拆分方案：**

Agent A（创作者）： 别管字数，别管文采，先把结构填满，把事说清楚。

Agent B（编辑）： 拿着 A 的稿子，只做减法，只做润色。

**📝【拆分示范】**

+:----------------------------------------------------------------------------------------+
| \[写作 Agent\] 输出： 一篇 2000 字的初稿，逻辑通顺但啰嗦。                              |
|                                                                                         |
| \[改写 Agent\] 指令： "保留原文所有观点，但把字数压缩到 800 字，把口吻改成'毒舌'风格。" |
+-----------------------------------------------------------------------------------------+

**5.3.3 拆分后的关键：设计"接力棒"**

很多人害怕拆分，是因为觉得"串起来好麻烦"。

其实，只要你定好"接力棒"，流程比单个 Agent 还要稳。

这里的秘诀是：上一个人的终点，必须是下一个人的起点。

你不需要复杂的编程，只需要在岗位说明书里加一个"交接字段"。

**📝【通用模板】接力棒规则**

+--------------------------------------------------------------+
| 上游 Agent（传棒者）的输出规定：                             |
|                                                              |
| 你的最后一行输出，必须包含一个 \[总结清单\]。                |
|                                                              |
| 方便下一个 Agent 直接读取。                                  |
|                                                              |
| 下游 Agent（接棒者）的输入规定：                             |
|                                                              |
| 你只允许读取上一步输出的 \[总结清单\]。                      |
|                                                              |
| 如果清单里缺信息，立刻报错，不要自己编。                     |
+--------------------------------------------------------------+

**【举个例子】**

+-----------------------------------------------------------------------+
| 搜集 Agent 最后必须输出一个：### 资料摘要清单                         |
|                                                                       |
| 写作 Agent 的第一句指令是：读取 \### 资料摘要清单，将其扩写为正文\... |
+-----------------------------------------------------------------------+

拆分不是为了把事情搞复杂，而是为了让每个环节都"简单到不会出错"。

**5.4 制作可直接复用的 Agent 岗位说明书**

你有没有发现，很多人用 AI 最大的问题是：一次性。今天调教得很好，过了一周忘了当时怎么说的，又得从头来过。这不叫工作流，这叫碰运气。真正的高手，手里都有一叠"岗位说明书（JD）"。下次要做同类任务，只需要把卡片拿出来，替换掉几个关键词，直接丢给 AI，它立马就能进入状态。

这一节我不教你写提示词，而是直接给你一张"万能填空表"。请把下面这个模板保存好，这是你未来所有 Agent 的原型。

**5.4.1 万能岗位说明书·填空版**

以后每创建一个新 Agent，请直接复制下面这段文字，填上你的内容。它包含了让 Agent 听话的所有核心指令。

**5.4.2 实例拆解：一张"及格"的岗位说明书长什么样？**

为了让你看清"普通指令"和"岗位说明书"的巨大差距，我们来做一个真实的 A/B 测试。

假设你的任务是：为一款"恒温咖啡杯"写一篇带货笔记。

**❌ 小白写法：**

"帮我写一篇小红书笔记，推荐一款恒温咖啡杯，要吸引人一点，带点 Emoji 表情。"

**💣 翻车后果：**

AI 味太重： 开头必定是"亲爱的宝子们！今天我要推荐\..."，一看就是机器写的。

重点跑偏： 它可能写了一堆"55度恒温技术参数"，却忘了写"冬天喝冷咖啡有多难受"这个痛点。

毫无转化： 结尾没有引导购买，只是一句空洞的"快去买吧"。

下面是一份基于 5.4.1 模板填写的完整说明书：

**5.4.3 绝不让文档吃灰：把说明书变成"即插即用"的武器**

写完岗位说明书（JD）只是第一步。很多人的痛点是：JD 写得很完美，但存在电脑深处的某个文件夹里。每次要用 AI 干活，光是"找文件 → 打开 → 复制 → 粘贴"这套动作，就得花 30 秒。

这 30 秒的摩擦力，足以毁掉你的工作流。 很多时候你嫌麻烦，干脆就"算了，我自己写吧"，于是 AI 又变回了摆设。真正的 AI 工作流，必须是零摩擦的。

这一节教你三招，把死板的文档变成随叫随到的"个人军火库"。

**第1招：由"文档"变为"咒语"（输入法快捷短语）**

这是最简单、但效率提升最恐怖的一招。

对于那些纯文字、高频使用的 Agent（比如：润色邮件、写周报、小红书文案），千万别存文档。请把它们存进你的输入法里。

+----------------------------------------------------------------------------------------------------------+
| **动作：** 打开搜狗/微信键盘/苹果自带输入法的"自定义短语"功能。                                          |
|                                                                                                          |
| **设置：**                                                                                               |
|                                                                                                          |
| **输入码：** /xhs（代表小红书 Agent）                                                                    |
|                                                                                                          |
| **输出内容：** 粘贴你那 800 字的完整岗位说明书。                                                         |
|                                                                                                          |
| **效果：**                                                                                               |
|                                                                                                          |
| 以后不管是在 ChatGPT 网页里，还是在手机 App 里，你只要敲下 /xhs 四个键，那位"百万粉丝博主"瞬间灵魂附体。 |
|                                                                                                          |
| 这叫"唤出式调用"，你把千军万马藏在了指尖下。                                                             |
+----------------------------------------------------------------------------------------------------------+

**第2招：由"文本"变为"员工"（Bot 平台封装）**

如果你需要的 Agent 很重，比如它需要"记住你公司的 50 个产品报价"，或者需要"一直模仿你的语气"，光靠粘贴文本就不够了。这时候，你要给它一个"永久的家"。这就是市面上那些 Agent 搭建平台（如 Coze、n8n、Dify、ChatGPT GPTs）的真正用法：它们是 JD 的"容器"。

+----------------------------------------------------------------------------------------------------------------------------+
| **动作：**                                                                                                                 |
|                                                                                                                            |
| 新建一个 Bot。                                                                                                             |
|                                                                                                                            |
| 把你的 **JD（人设与目标）** 粘贴进"提示词"区。                                                                             |
|                                                                                                                            |
| 把你整理好的 **产品手册 PDF** 上传到"知识库"区。                                                                           |
|                                                                                                                            |
| 点击发布。                                                                                                                 |
|                                                                                                                            |
| **效果：**                                                                                                                 |
|                                                                                                                            |
| 你拥有了一个永远记得你公司底细的专属员工。                                                                                 |
|                                                                                                                            |
| 你不需要每次都重复"我是谁、卖什么的"，上来直接派活："给我写个某产品的促销案"，它马上就能调用知识库里的报价，且绝对不会错。 |
+----------------------------------------------------------------------------------------------------------------------------+

**第3招：由"迭代"变为"补丁"（错题本机制）**

也没必要像程序员一样搞什么 v1.0、v2.0 的版本号，那是给自己找麻烦。JD 是需要"养"的，而养它的最好方式是"打补丁"。当你的 Agent 犯错时（比如它总是胡编乱造数据），绝对不要在对话框里临时纠正它，因为你这次纠正了，下次新建对话它还是会忘。正确的"打补丁"流程：

+:--------------------------------------------------------------------------------------------------------+
| **Step 1 发现 Bug：**比如你发现它写的小红书文案里，总喜欢用"绝绝子"这个词，太土了。                     |
|                                                                                                         |
| **Step 2 追杀回老巢：**立刻打开你的 JD 原文（或 Bot 配置页）。                                          |
|                                                                                                         |
| **Step 3 写入禁令（打补丁）：**找到 【限制红线】 这一栏，加一条：禁止使用"绝绝子"等过时词汇，违者重写。 |
|                                                                                                         |
| **Step 4 保存退出。**从此以后，这个错误永远不会再出现。好用的 Agent，都是靠这样一个个"补丁"堆出来的。   |
+---------------------------------------------------------------------------------------------------------+
